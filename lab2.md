## Шаг 1. Стратегия валидации и воспроизводимость

### Стратегия валидации
Для задачи **непрерывного Keyword Spotting** в аудиопотоке (а не дискретных команд) выбрана **speaker-independent (speaker-disjoint) валидация** с учётом sliding-window подхода.

**Обоснование:**  
- Google Speech Commands Dataset содержит короткие записи (1 секунда), но в реальной системе ESP32 передаёт **постоянный аудиопоток** (или поток log-mel спектрограмм).  
- Модель обучается на 1-секундных окнах с перекрытием (overlap 50 %), чтобы корректно работать на непрерывном стриме.  
- Официальные сплиты датасета (`validation_list.txt` и `testing_list.txt`) гарантируют, что **все записи одного спикера** находятся только в одном сплите (train / validation / test). Это полностью исключает утечку по спикеру.  
- Дополнительно: **stratified 5-fold cross-validation** внутри train-сета + оценка на валидационном и тестовом сете с метрикой **macro F1-score** и **false-positive rate per hour** (критично для стриминга).

**Проверка отсутствия переобучения:**  
Сравнение метрик на validation и test (разница ≤ 3–5 %). Дополнительно измеряем **false alarm rate** на длинных тестовых записях без ключевых слов.

### Компоненты, подлежащие версионированию для воспроизводимости
Для полной воспроизводимости экспериментов версионируются:

| Компонент                  | Инструмент версионирования       | Что версионируется                                      | Почему обязательно |
|----------------------------|----------------------------------|---------------------------------------------------------|---------------------|
| Данные                     | DVC + Git                        | Исходные WAV + sliding-window сплиты + синтетические стримы | Изменение окон/overlap меняет результат |
| Код и пайплайн             | Git + теги                       | Скрипты обучения, inference streaming, конфиги          | Полная воспроизводимость |

## Шаг 2. Анализ утечек данных

Утечки данных **маловероятны**, но проанализированы с учётом стриминговой природы системы.

### Потенциальные источники утечек и меры предотвращения

1. **Speaker leakage**  
   **Риск:** Модель запоминает голос спикера вместо семантики команды.  
   **Мера:** Только официальные speaker-disjoint сплиты Google. Перед обучением — автоматическая проверка пересечения speaker_id.

2. **Leakage через предобработку**  
   **Риск:** Статистики нормализации (mean/std) вычисляются по всему потоку до сплита.  
   **Мера:** Нормализация считается **только на train-сете**. Параметры сохраняются и применяются к val/test/stream.

3. **Leakage через аугментацию в стриме**  
   **Риск:** Аугментированные окна из test попадают в train.  
   **Мера:** Аугментация (SpecAugment, background noise) применяется **on-the-fly только в train** внутри DataLoader.

4. **Temporal leakage**  
   **Риск:** Окна с перекрытием «видят» будущие данные.  
   **Мера:** При разбиении на train/val/test перекрытие учитывается только внутри одного сплита.

**Вывод:** При speaker-disjoint сплите и DVC-пайплайне утечки исключены. Для собственного датасета добавим метаданные speaker_id и device_id.

## Шаг 3. Масштабирование

### Оценка ожидаемой нагрузки (с учётом непрерывного стрима)
- **Количество устройств:** 500–2000 активных ESP32 (реалистичный пилот умного дома).  
- **Характер нагрузки:** **не дискретные запросы**, а **постоянные стримы** от каждого устройства.  
  - ESP32 передаёт аудио-чанки (или log-mel спектрограммы) каждые 100–250 мс (скользящее окно 1 секунда с overlap 50 %).  
- **Средняя нагрузка:** 500–2000 одновременных стриминговых сессий.  
- **Пиковая нагрузка:** до 3000 сессий (все устройства активны одновременно).  
- **Трафик:**  
  - Cloud-only (сырой PCM 16 кГц): ~16–32 KB/сек на устройство.  
  - Hybrid (log-mel спектрограмма): ~2–4 KB/сек на устройство → экономия 85–90 %.  
- **Допустимая задержка:** **end-to-end latency ≤ 300 мс** от момента произнесения слова до получения JSON-ответа на ESP32.

### Архитектурные паттерны масштабирования
1. **Долгоживущие стриминговые соединения**  
   Ingestion Gateway использует **WebSocket** или **gRPC bidirectional streaming** (или MQTT 5.0 с QoS 1). Каждое устройство держит одно постоянное соединение.

2. **Горизонтальное масштабирование**  
   Несколько реплик ML Inference Engine за балансировщиком (Kubernetes HPA по CPU/RPS или количеству активных стримингов).

3. **Асинхронная обработка + очередь**  
   Входящие чанки попадают в **Redis Stream** / Kafka. ML-движок потребляет их асинхронно, детектит ключевое слово и пушит JSON обратно по тому же каналу.

4. **Кэширование и early-exit**  
   Redis Cache для часто повторяющихся спектрограмм (TTL 30 сек). Если confidence > 0.97 — мгновенный ответ без полного инференса.

5. **Edge-offloading по умолчанию**  
   Основной режим — **hybrid**: ESP32 сам извлекает log-mel и отправляет только признаки. Это снижает нагрузку на облако в 8–10 раз.

6. **Мониторинг**  
   Prometheus + Grafana отслеживают количество активных стримингов, latency per stream и false-positive rate.

**Вывод:** Система масштабируется до тысяч одновременных стриминговых сессий за счёт hybrid-режима, горизонтального масштабирования и асинхронных очередей.

## Шаг 4. Корректировка архитектуры системы и стека

На основе Шагов 1–3 и понимания, что **ESP32 передаёт непрерывный поток (stream), а не дискретные команды**, внесены следующие изменения в документацию ЛР1 (шаги 5–7):

### 4.1 Изменения в высокоуровневой архитектуре (Шаг 5 ЛР1)
- Добавлен **Streaming Ingestion** (WebSocket / gRPC / MQTT).  
- Добавлены **Redis Stream / Kafka** для буферизации чанков.  
- ML Inference Engine теперь работает в режиме **continuous sliding-window inference**.  
- Ответ JSON пушится обратно по тому же стриминговому каналу только при обнаружении ключевого слова (или с заданным интервалом).

### 4.2 Изменения в модулях и протоколах (Шаг 6 ЛР1)
- **Ingestion Gateway** теперь поддерживает долгоживущие стриминговые сессии.  
- **Протокол AFE ↔ Gateway:** WebSocket / gRPC bidirectional streaming (или MQTT publish/subscribe).  
- **Протокол Gateway ↔ ML Engine:** внутренний асинхронный (Redis Stream).  
- **Ответ ESP32:** JSON пушится только при срабатывании (`{"predicted_class": "red", "confidence": 0.94, "timestamp": "..."}`).

### 4.3 Обновлённый стек технологий (Шаг 7 ЛР1)
**Дополнения и корректировки:**

- **Ingestion Gateway:** FastAPI + **WebSocket** / **gRPC** + **Redis Stream**  
- **ML Inference Engine:** PyTorch + torchaudio + **streaming inference** (sliding window)  
- **Очередь и буферизация:** **Redis 7.x** (Streams) или Kafka  
- **Версионирование и эксперименты:** MLflow + DVC (без изменений)  
- **Мониторинг:** Prometheus + Grafana (добавлено)

**Почему добавлено:**  
- WebSocket/gRPC необходимы для непрерывного стрима вместо HTTP POST.  
- Redis Stream — для обработки тысяч параллельных потоков без блокировки.  

Все предыдущие обоснования (PyTorch, Docker, S3 и т.д.) остаются актуальными.

**Итог корректировки:** Архитектура полностью адаптирована под **режим постоянного аудиопотока** от ESP32, стала воспроизводимой, защищённой от утечек и готовой к масштабированию до тысяч устройств при сохранении latency ≤ 300 мс.
