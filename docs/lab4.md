## Шаг 1. Разработка инференс-модуля

На этом шаге реализован отдельный микросервис предсказаний в директории `app` на базе `FastAPI`.  
Сервис загружает финальный чекпоинт модели (`ResNetKWS`) и предоставляет API для офлайн- и realtime-инференса.

### 1.1. Что делает модуль `app`

Инференс-модуль отвечает за полный путь запроса:

1. приём аудиоданных через REST/WebSocket;
2. приведение сигнала к формату модели (моно, целевая частота 16 кГц, окно 1 секунда);
3. запуск модели и получение вероятностей классов;
4. формирование ответа (`predicted_class`, `confidence`, `top_k`, `latency_ms`);
5. логирование запроса и экспорт метрик.

Ключевые эндпоинты:

- `POST /predict` - одиночный прогноз по аудиофайлу;
- `POST /predict-base64` - одиночный прогноз по base64-пейлоаду;
- `POST /predict-stream` - скользящее окно по записи на сервере;
- `WS /ws/kws` - realtime-поток с микрофона;
- `GET /health`, `GET /ready`, `GET /metrics` - сервисные и мониторинговые эндпоинты.

### 1.2. Два режима инференса

В системе реализованы два рабочих режима:

1. **Одиночный (batch) режим**  
   Для готового WAV/аудиофрагмента выполняется один вызов модели и возвращается результат классификации.

2. **Потоковый (streaming) режим**  
   Аудио обрабатывается по скользящему окну 1 секунда с заданным шагом (`stride`), порогом уверенности и рефрактерным периодом для подавления повторных срабатываний.

Такой подход позволяет использовать одну и ту же модель как для пост-анализа записей, так и для детекции команд в реальном времени.

### 1.3. Поддержка входных представлений: raw PCM и log-mel

Инференс-контур спроектирован для двух типов входа:

- **`raw PCM`** (основной текущий путь): сервис принимает сырой аудиосигнал (в файле или в виде `float32`-чанков по WebSocket), после чего сам выполняет необходимую подготовку перед моделью;
- **`log-mel`** (гибридный путь): допускается предварительное вычисление признаков на клиентской/edge-стороне с передачей в сервис уже компактного признакового представления.

В текущей реализации основной рабочий поток построен вокруг `raw PCM`, а преобразование в `log-mel` выполняется в модельном пайплайне (`ResNetKWS`) автоматически. Это сохраняет совместимость с гибридным сценарием и упрощает дальнейшее расширение сервиса.

### 1.4. Результат шага

В результате шага 1 получен работоспособный инференс-микросервис, который:

- использует финальную модель из ЛР3;
- поддерживает batch и streaming сценарии;
- работает с аудио в формате, пригодном для realtime-KWS;
- готов к интеграции с UI, хранилищем логов и подсистемой мониторинга на следующих шагах ЛР4.

## Шаг 2. Интеграция компонентов

На этом этапе все разработанные модули объединены в единую рабочую систему:

- `ui` (Streamlit) выступает клиентом и отправляет запросы в `app` (FastAPI) по REST и WebSocket;
- `app` выполняет инференс финальной модели и возвращает результат в интерфейс;
- журнал запросов сохраняется в SQLite через `RequestLogStore`;
- метрики публикуются в формате Prometheus через эндпоинт `/metrics`;
- служебные эндпоинты `/health` и `/ready` обеспечивают проверку состояния сервиса.

Таким образом, вместо набора изолированных частей сформирован единый контур:  
**пользовательский интерфейс -> API-инференс -> логирование/мониторинг**.

### 2.1. Что изменилось относительно предварительной архитектуры ЛР1 (шаги 5-7)

В процессе интеграции уточнены и частично скорректированы компоненты, описанные в ЛР1:

| Было в предварительном описании | Стало в итоговой системе |
|---|---|
| Общий контур приёма запросов без жёсткой фиксации протоколов | Зафиксирован `FastAPI + Uvicorn` с двумя каналами: REST (`/predict`, `/predict-base64`, `/predict-stream`) и WebSocket (`/ws/kws`) |
| Инференс как единый блок без явного разделения сценариев | Явно разделены два режима: batch (одиночные запросы) и streaming (скользящее окно в realtime) |
| Мониторинг на уровне идеи | Реализован практический минимум: `/metrics` + счётчики/гистограммы запросов и латентности |
| Аналитическое хранилище как абстрактный компонент | На лабораторном контуре применено лёгкое хранилище SQLite для фактического журнала инференс-запросов |
| Связь UI и backend описана концептуально | Интеграция выполнена по факту: страница realtime в `ui` управляет параметрами потока и отображает предсказания сервиса |
| Версионирование модели не было частью ответа API | В ответы и логи добавлено поле `model_version` для трассировки и воспроизводимости |

### 2.2. Итог шага

Итоговая архитектура стала более конкретной и эксплуатационно ориентированной:  
интерфейс, инференс, хранение логов и мониторинг связаны в единую систему с чёткими API-контрактами и проверяемым состоянием сервиса.

## Шаг 3. Внедрение базового мониторинга

На текущем этапе в сервис уже добавлены базовые механизмы наблюдаемости, которые позволяют контролировать его состояние и нагрузку.

### 3.1. Логирование запросов

Реализованы два уровня логирования:

- **технические логи приложения** через стандартный `logging` (успешные и ошибочные вызовы API, сообщения об исключениях);
- **журнал инференс-запросов в БД (SQLite)** через `RequestLogStore` с сохранением ключевых полей:
  - `created_at`,
  - `filename`,
  - `predicted_class`,
  - `confidence`,
  - `latency_ms`,
  - `model_version`.

Это даёт возможность как онлайн-диагностики ошибок, так и последующего анализа качества работы сервиса на реальных запросах.

### 3.2. Метрики доступности системы

Для контроля доступности реализованы сервисные эндпоинты:

- `GET /health` - быстрый признак «процесс жив» (`status: ok`);
- `GET /ready` - признак готовности к работе (`status: ready`), включая информацию о загруженной версии модели и доступных классах.

Разделение `health` и `ready` позволяет отдельно контролировать факт запуска процесса и факт готовности именно ML-компонента.

### 3.3. Сбор метрик нагрузки на систему (связь с ЛР2, шаг 3)

Введены метрики, отражающие нагрузку и производительность:

- `kws_requests_total{endpoint,status}` - счётчик запросов по эндпоинтам и статусам (ok/bad_request/error);
- `kws_request_latency_ms{endpoint}` - гистограмма полной латентности HTTP-запросов;
- `kws_inference_latency_ms` - гистограмма «чистого» времени инференса модели.

Эти метрики покрывают базовые показатели из ЛР2 (шаг 3): интенсивность запросов, задержки обработки и стабильность сервиса под нагрузкой.

### 3.4. Экспорт метрик для мониторинга

Экспорт реализован через эндпоинт:

- `GET /metrics` - отдаёт метрики в формате Prometheus (`text/plain`), пригодном для сбора внешней системой мониторинга.

Таким образом, текущая версия сервиса уже готова к подключению стандартного стека наблюдаемости (например, Prometheus + Grafana) без изменения бизнес-логики.

### 3.5. Итог шага

Базовый мониторинг внедрён:  
логирование запросов работает, доступность контролируется через `health/ready`, нагрузка измеряется гистограммами и счётчиками, а метрики экспортируются через `/metrics` для внешнего мониторинга.

## Шаг 4. Контейнеризация и оркестрация

На финальном этапе в Docker-контейнер упакован инференс-сервис (API). Пользовательский интерфейс запускается отдельно на хосте и используется как example-клиент для проверки API.

### 4.1. Dockerfile для функциональных компонентов

Для каждого ключевого компонента используется отдельный Dockerfile:

- `Dockerfile.inference` - контейнер инференс-сервиса (`FastAPI + Uvicorn`):
  - установка зависимостей из `requirements.txt`;
  - копирование модулей `app`, `models`, `artifacts`;
  - открытие порта `8000`;
  - старт командой `uvicorn app.main:app --host 0.0.0.0 --port 8000`.

- `Dockerfile.ui` - контейнер пользовательского интерфейса (`Streamlit`):
  - установка зависимостей из `requirements.txt`;
  - копирование директории `ui`;
  - открытие порта `8501`;
  - старт командой `streamlit run ui/app.py --server.port=8501 --server.address=0.0.0.0`.

### 4.2. Оркестрация в `docker-compose.yml`

В `docker-compose.yml` описан контейнер инференс-сервиса и параметры его запуска:

- сервис `app`:
  - сборка из `Dockerfile.inference`;
  - переменные окружения: `MODEL_PATH`, `DB_PATH`;
  - тома:
    - `inference_data:/data` для сохранения SQLite БД между перезапусками;
    - `./artifacts:/app/artifacts:ro` для подключения артефакта модели в read-only режиме;
  - публикация порта `8000:8000`;
  - `healthcheck` по эндпоинту `/health`.

- том `inference_data`:
  - обеспечивает сохранение данных логирования (SQLite) при перезапуске контейнеров.

Интерфейс `Streamlit` запускается отдельно от `docker-compose` и обращается к API по адресу `http://localhost:8000` (через переменную `API_URL`).

### 4.3. Запуск одной командой

Запуск API в Docker выполняется одной командой:

```bash
docker compose up --build
```

После запуска контейнера доступны:

- API: `http://localhost:8000/docs`.

UI запускается отдельно (на хосте) и используется как пример/демо для проверки API:

```bash
pip install -r requirements.txt
streamlit run ui/app.py --server.port 8501
```

В `README.md` зафиксированы:

### 4.5. Итог шага

Контейнеризация завершена для основного сервиса: API запускается в Docker, данные логирования сохраняются в отдельном томе, а UI запускается отдельно как example-интерфейс для проверки и демонстрации работы API. Это не противоречит заданию, так как UI не является обязательной частью production-развёртывания сервиса.
