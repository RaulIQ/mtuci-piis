# Лабораторная работа №3  
**Тема:** разработка ML-модели для KWS (Keyword Spotting) в контексте проектируемого сервиса

## 1. Цель работы
Получить практические навыки разработки ML-модели в контексте проектируемой системы:
- выбор и реализация baseline-модели;
- проведение экспериментов с логированием параметров и метрик;
- анализ ошибок модели и итеративное улучшение;
- подготовка финальной модели для внедрения в сервис.

## Шаг 1. Подготовка данных

### 1.1. Источник данных
В качестве исходного набора данных использован `torchaudio.datasets.SPEECHCOMMANDS` (Speech Commands v0.02), выбранный в ЛР1.

Для соблюдения стратегии валидации из ЛР2 использованы официальные speaker-disjoint разбиения датасета:
- `training` — 84843 записей;
- `validation` — 9981 запись;
- `testing` — 11005 записей.

Это гарантирует отсутствие пересечения спикеров между обучением, валидацией и тестом.

### 1.2. Целевые классы и формирование выборок
Целевые команды:  
`one, two, three, four, five, six, seven, eight, nine`  
Дополнительно сформированы классы:
- `unknown` — стратифицированная выборка из остальных 26 команд;
- `silence` — фрагменты тишины длиной 1 секунда.

Количество `unknown` и `silence` для каждого split выбрано равным размеру максимального базового класса в соответствующем split, чтобы уменьшить дисбаланс.

Итоговые размеры подготовленных выборок:
- `train`: 34388;
- `val`: 4033;
- `test`: 4579.

### 1.3. Предобработка
Для каждого аудио-примера выполняется единый пайплайн:
1. Загрузка WAV (mono).
2. Приведение частоты дискретизации к `16 kHz` (resample при необходимости).
3. Приведение длительности к 1 секунде:
   - если короче — zero-padding;
   - если длиннее — обрезка до первых 16000 сэмплов.
4. Обработка специальных случаев:
   - для `silence` при отсутствии исходного файла используется нулевой сигнал длиной 1 секунда;
   - для сохраненных/перенесенных путей реализована нормализация пути к аудиофайлу.

### 1.5. Фиксация предобработки в коде
Предобработка реализована и зафиксирована в ноутбуке `lab3.ipynb` отдельными функциями:
- `collect_subset_paths(...)` — выбор целевых классов;
- `collect_unknown_paths(...)` — формирование класса `unknown`;
- `make_silence_rows(...)` — генерация класса `silence`;
- `extract_features(...)` — извлечение MFCC-признаков;
- `build_xy(...)` — сбор `X/y` и кэширование (`.npy`) для воспроизводимости.

Для ускорения повторных запусков используются кэши:
- `artifacts/train_X.npy`, `artifacts/train_y.npy`;
- `artifacts/val_X.npy`, `artifacts/val_y.npy`;
- `artifacts/test_X.npy`, `artifacts/test_y.npy`.

---
**Итог Шага 1:** данные загружены, предобработаны и разделены на `train/val/test` в соответствии со стратегией ЛР2; подготовлен воспроизводимый код и артефакты для следующего шага (обучение baseline-модели).


## Шаг 2. Выбор и реализация baseline-модели

### 2.1. Выбор baseline
Задача ЛР3 — многоклассовая классификация речевых команд (KWS), поэтому в качестве baseline выбрана **логистическая регрессия**.

Выбор обоснован тем, что модель:
- проста в реализации и интерпретации;
- быстро обучается;
- задает «нижнюю границу» качества, с которой сравниваются более сложные подходы.

### 2.2. Реализация обучения
Обучение baseline реализовано в ноутбуке `lab3.ipynb`.

Использован пайплайн `scikit-learn`:
- `StandardScaler()` для нормализации признаков;
- `LogisticRegression(max_iter=1000, random_state=SEED)` как классификатор.

Обучение выполняется на `X_train, y_train`, оценка — на `X_val, y_val`:
- основная метрика качества: `macro F1`;
- дополнительно формируется `classification_report` (precision/recall/F1 по каждому классу).

### 2.3. Метрики baseline
Зафиксированные результаты baseline на валидационной выборке:
- `macro F1 (val) = 0.6577`;
- `accuracy (val) = 0.6650`.

Ключевые наблюдения по отчету классификации:
- класс `silence` распознается лучше остальных (`F1 ≈ 0.9687`);
- наибольшие ошибки у класса `unknown` (`F1 ≈ 0.2340`), что ожидаемо из-за высокой вариативности этого класса.

### 2.4. Сохранение результатов
Метрики baseline и полный `classification_report` сохранены в выводе ноутбука `lab3.ipynb` и используются как точка отсчета для следующих экспериментов (Шаги 3-4).

---
**Итог Шага 2:** baseline-модель (логистическая регрессия) выбрана, обучена и оценена; получены и зафиксированы метрики качества для дальнейшего сравнения с улучшенными моделями.

## Шаг 3. Настройка экспериментального пайплайна

### 3.1. Цель шага
На данном этапе организовано воспроизводимое логирование экспериментов, чтобы:
- сравнивать модели в едином формате;
- фиксировать параметры запуска и итоговые метрики;
- выбирать лучшую конфигурацию на основе валидации.

### 3.2. Реализация логирования
Экспериментальный пайплайн реализован в `lab3.ipynb`.

Базовая структура:
- создается список `experiments = []` для накопления результатов;
- используется функция `log_experiment(name, model, X_tr, y_tr, X_va, y_va, params=None)`.

Функция `log_experiment(...)` выполняет:
1. обучение модели на `train`;
2. предсказание на `val`;
3. расчет метрики `f1_macro` (основной KPI);
4. сохранение записи в лог экспериментов:
   - `name` (название эксперимента/модели),
   - `params` (гиперпараметры),
   - `val_macro_f1` (качество на валидации).

### 3.3. Организация серии экспериментов
Для сравнения моделей используется единый протокол:
- логистическая регрессия с подбором параметра `C` через `GridSearchCV`;
- `RandomForest` с подбором `n_estimators`, `max_depth`, `min_samples_split` через `GridSearchCV`;
- единая функция оценки и логирования для всех запусков.

После выполнения запусков формируется таблица:
- `exp_df = pd.DataFrame(experiments).sort_values("val_macro_f1", ascending=False)`.

Такой формат позволяет быстро ранжировать эксперименты по качеству и прозрачно выбирать лучшую модель.

### 3.4. Зафиксированные артефакты шага
В рамках шага сохранены:
- код пайплайна логирования в `lab3.ipynb`;
- журнал экспериментов в виде таблицы `exp_df` (параметры + метрики);
- единая метрика сравнения `macro F1` на `val`.

---
**Итог Шага 3:** настроен экспериментальный пайплайн с централизованным логированием параметров и метрик, что обеспечивает воспроизводимость и корректное сравнение моделей в последующих шагах.

## Шаг 4. Проведение экспериментов

### 4.1. Стратегия улучшения baseline
Серия экспериментов строилась по принципу поэтапного усложнения:
1. **простые ML-модели** на engineered-признаках (MFCC + статистики);
2. переход к **CNN** на 2D log-Mel спектрограммах;
3. переход от CNN к более глубокой архитектуре **ResNet**.

Такой порядок позволил получить прозрачную траекторию роста качества модели.

### 4.2. Этап 1 — простые модели (классический ML)
Проведены эксперименты с:
- `LogisticRegression` (baseline и tuned-версия);
- `RandomForest` (tuned-версия).

Что варьировалось:
- алгоритм (`LogReg` vs `RandomForest`);
- гиперпараметры (`C` для логистической регрессии, `n_estimators/max_depth/min_samples_split` для леса);
- единая схема оценки (`macro F1` на `val`).

Логирование:
- каждый запуск фиксировался через `log_experiment(...)`;
- результаты собирались в `experiments`;
- формировалась сводная таблица `exp_df` с сортировкой по `val_macro_f1`.

Ключевой ориентир этапа:
- baseline `LogReg`: `macro F1 (val) = 0.6577`.

### 4.3. Этап 2 — переход к CNN
После предела качества простых моделей выполнен переход к нейросетевому подходу:
- признаки изменены с MFCC-статистик на **2D log-Mel**;
- реализована компактная архитектура `SmallKwsCNN` (`models/small_kws_cnn.py`);
- обучение проведено в `lab3.ipynb` по эпохам с контролем `best val macro-F1`.

Зафиксированные результаты CNN:
- лучший `val F1_macro = 0.7801`;
- `test F1_macro = 0.7643`;
- модель сохранена в `artifacts/kws_cnn.pt`.

Логирование этапа CNN:
- для каждой эпохи выводились `loss`, `acc`, `F1_macro`, `F1_weighted`;
- отдельно сохранялся лучший чекпоинт по валидации;
- строились `classification_report` и `confusion matrix`.

### 4.4. Этап 3 — переход от CNN к ResNet
Для дальнейшего роста качества выполнен переход к более мощной модели:
- использована архитектура `ResNetKWS` (ResNet-18 для KWS);
- сохранена совместимая схема данных (`train/val/test` из Шага 1);
- обучение и выбор лучших весов выполнялись по `best_val_f1`.

Зафиксированные результаты ResNet (`lab3_resnet.ipynb`):
- `Validation acc = 0.9574`, `F1_macro = 0.9577`;
- `Test acc = 0.9559`, `F1_macro = 0.9563`;
- веса сохранены в `artifacts/kws_resnet.pt`.

### 4.5. Логирование экспериментов (итог)
Требование «каждый эксперимент должен быть залогирован» выполнено на всех уровнях:
- **классические модели:** журнал `experiments` + таблица `exp_df` (параметры и `val_macro_f1`);
- **CNN:** поэпоховые логи метрик + отчеты на `val/test` + сохраненный артефакт;
- **ResNet:** поэпоховые логи, выбор лучших весов по валидации, финальные метрики на `test`, сохраненный артефакт.

---
**Итог Шага 4:** проведена последовательная серия экспериментов с переходом от простых моделей к CNN и далее к ResNet; на каждом этапе зафиксированы параметры, метрики и итоговые артефакты, что обеспечивает воспроизводимость и обоснованный выбор финальной модели.

## Шаг 5. Анализ ошибок

### 5.1. Какие модели анализировались
Детальный анализ ошибок выполнен для:
- `SmallKwsCNN` (как промежуточной нейросетевой модели);
- `ResNetKWS` (как лучшей итоговой модели).

Это позволило понять, какие проблемы были у CNN и какие из них снимаются при переходе к ResNet.

### 5.2. Материалы анализа (графики и отчеты)
Для каждой модели использованы следующие инструменты анализа:
- `classification_report` (precision/recall/F1 по каждому классу);
- `confusion matrix` (матрица ошибок);
- таблица/примеры misclassification (ошибочно распознанные примеры).

Графики и таблицы построены в ноутбуках:
- `lab3.ipynb` — для baseline/CNN (включая `CNN confusion matrix (test)` и примеры ошибок);
- `lab3_resnet.ipynb` — для ResNet (`ResNet confusion matrix (test)`).

### 5.3. Наблюдения по слабым местам
#### CNN
Ключевые метрики:
- `CNN macro F1 (test) = 0.7643`;
- для класса `unknown`: `F1 = 0.3799` (test), `F1 = 0.3943` (val);
- для `silence`: почти идеальное качество (`F1 ≈ 0.998` на test).

Слабое место CNN: распознавание класса `unknown` и похожих по звучанию коротких команд.  
В примерах misclassification встречаются ошибки вида `eight -> silence` и `eight -> unknown`.

#### ResNet (лучшая модель)
Ключевые метрики:
- `Validation: acc = 0.9574, F1_macro = 0.9577`;
- `Test: acc = 0.9559, F1_macro = 0.9563`.

По классам:
- `silence`: `F1 = 0.9989` (test);
- `unknown`: `F1 = 0.8695` (test), что значительно лучше CNN.

Вывод: переход к ResNet резко уменьшил число ошибок по «трудным» классам, особенно по `unknown`.

### 5.4. Причины ошибок
Основные причины, выявленные по отчетам и матрицам:
- **высокая внутриклассовая вариативность `unknown`** (26 разных слов в одном классе);
- **фоновые шумы и различия условий записи** (микрофон, расстояние, громкость);
- **фонетическая близость команд** (часть цифровых слов путается между собой);
- **ограниченная выразительность компактной CNN** для сложного акустического пространства.

### 5.5. Предложенные способы улучшения
Для дальнейшего снижения ошибок предлагаются:
- усилить аугментации аудио: шум, реверберация, time-shift, random gain;
- ввести hard-negative mining для `unknown` (добавлять наиболее «похожие» ложные срабатывания);
- откалибровать пороги уверенности и правило reject для uncertain-предсказаний;
- расширить контроль по устройствам/доменам записи (domain shift);
- проверить distillation/ensemble (ResNet teacher -> компактный student для real-time).

### 5.6. Итог анализа ошибок
Анализ показал, что главный источник ошибок в KWS-постановке — класс `unknown` и близкие по акустике команды.  
Эволюция моделей `классический ML -> CNN -> ResNet` дала последовательное улучшение устойчивости, а ResNet обеспечила наилучший компромисс по качеству для внедрения в сервис.

---
**Итог Шага 5:** выполнен детальный анализ ошибок по отчетам, матрицам и примерам misclassification; выявлены слабые места и причины ошибок, сформирован набор практических мер для дальнейшего улучшения модели.

## Шаг 6. Выбор финальной модели

### 6.1. Выбранная финальная модель
По результатам серии экспериментов и анализа ошибок финальной выбрана модель **`ResNetKWS`**.

Причины выбора:
- лучший уровень качества среди всех протестированных подходов;
- существенное снижение ошибок по критичному классу `unknown`.

### 6.2. Достигнутые метрики
Итоговые метрики финальной модели (`lab3_resnet.ipynb`):
- **валидация:** `acc = 0.9574`, `F1_macro = 0.9577`, `F1_weighted = 0.9576`;
- **тест:** `acc = 0.9559`, `F1_macro = 0.9563`, `F1_weighted = 0.9558`.

Для сравнения с предыдущими этапами:
- baseline `LogReg`: `F1_macro (val) = 0.6577`;
- `SmallKwsCNN`: `F1_macro (test) = 0.7643`;
- `ResNetKWS`: `F1_macro (test) = 0.9563` (лучший результат).

### 6.3. Компромиссы (сложность vs качество)
Принятый компромисс:
- **качество:** ResNet значительно превосходит простые модели и CNN по устойчивости и точности;
- **сложность:** архитектура тяжелее в обучении и сопровождении, чем `LogReg`/малый CNN;
- **инференс:** стоимость одного предсказания выше, чем у baseline, но оправдана существенным приростом качества.

Практический вывод: для production-KWS в этом проекте приоритет отдан качеству распознавания (особенно по `unknown`) при умеренном увеличении вычислительной нагрузки.

### 6.4. Соответствие нефункциональным требованиям
#### Размер модели
- сохраненный артефакт финальной модели: `artifacts/kws_resnet.pt`;
- размер файла: **~43 MB**.

#### Задержка (latency) и пригодность для сервиса
- инференс-сервис измеряет и логирует latency (`inference_ms`, `request latency`) в `app/main.py`;
- в real-time режиме используется скользящее окно 1 сек и настраиваемый `poll_interval_sec` в диапазоне `0.05..1.0` сек (`app/ws_kws.py`);
- предусмотрены прикладные ограничения для стабильной работы в потоке (`confidence_threshold`, `refractory_sec`).

Таким образом, модель технически подготовлена к онлайн-инференсу и контролю SLA по задержке на уровне сервиса.

### 6.5. Сохранение модели для инференс-модуля
Финальная модель сохранена в формате PyTorch checkpoint:
- путь: `artifacts/kws_resnet.pt`;
- формат: словарь с `state_dict` и метаданными (`labels`, параметры спектрограммы и архитектуры);
- загрузка реализована в `KwsInferenceService` (`app/model.py`) через `torch.load(...)` и `model.eval()`.

---
**Итог Шага 6:** финальная модель `ResNetKWS` выбрана и обоснована по метрикам и анализу ошибок, соответствует требованиям внедрения в сервис и сохранена в формате, пригодном для прямой загрузки в инференс-модуль.
